# R2-CBBRTL · 结构优先 · CBB复用 · 专家判断蒸馏

**北极星** 结构是因，复用是果——把结构搞对，复用自然落位

**内核** 新需求 = 旧 proven 项目的一个 delta；顺「标注」把 delta 变成有界·可追·可验的变更，只改变受影响的，其余继承 proven 的正确

**铁律** 结构优先 · 接口即协议（边界=团队）· 极简优先

## TRACK A 底座：标注 = proven 资产上的 4 张映射（持续建，回库越用越肥）

| # | 映射 | 说明 | 标签 |
|---|------|------|------|
| 01 | 特性映射 | 需求 ↔ feature point ↔ 代码 | 需求 / feature point / 代码 |
| 02 | 设计模式 | 每块用哪个结构模具（FSM / 握手 / FIFO / CDC …） | FSM / 握手 / FIFO / CDC |
| 03 | 架构决策 | 为什么这么设计 = 不可破坏的约束 | 时序/延迟 · 吞吐/带宽 · 一致性/顺序 · 安全/隔离 |
| 04 | 验证环境映射 | feature point ↔ testbench / golden / scoreboard | testbench / golden / scoreboard |

↓ 喂 Agent  ↑ 回库 新 FP / 代码 / 模式 / 判断

## TRACK B 执行：一个新需求进来，顺这 4 张映射走

| 问题 | 动作 | 收益 |
|------|------|------|
| 改哪里？ | ① 定位 受影响的 FP + 代码行 | ✓ 精确定位，不大改 |
| 怎么写对？ | ② 改动落进已知正确的模具（结构错从源头堵死） | ✓ 站在 proven 结构上写，更小更稳 |
| 不破坏啥？ | ③ 守住架构约束 / 不变量 | ✓ 不破坏系统整体正确性 |
| 怎么重验？ | ④ 复用验证环境，只重跑受影响的 FP | ✓ 最快反馈，最少验证成本 |
| 结果 | → 没碰的代码原封不动、继承 proven 正确；碰到的小、可写对、当场可验 | ✓ 变更有界 · 可追 · 可验 |

## 流水线

残差（库里没有可改的那一小块）→ 高层建模 golden → RTL → 逐拍验证（对齐行为）→ 形式等价兜底

## 铁则

标注覆盖到的 = 被定义的“正确”；覆盖不到的 = bug 藏身处

→ 标注完整性 = 正确性上限
