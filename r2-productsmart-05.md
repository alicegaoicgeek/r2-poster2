# 双流水线，两个接口：产品流水线与工程流水线闭环成一条完整价值流

R2-ProductSmart

## 一、全景

产品流水线(产品敏捷)：企业战略→愿景→产品战略→发布计划→Sprint计划→每日计划

- 前向接口 ① Enabling Spec（向下）
- 反向接口 ② 工程信号回流（向上）

工程流水线(工程敏捷)：L1意图声明→L2可执行契约→L3验证架构→L4质量内建→L5工程效能(四里程碑门禁)

产品流水线侧：Peter + r2-productsmart（任务跟踪 · 状态持久化 · 轨迹管理）  
工程流水线侧：Fred + R2-harness（编排 · 扇出 · while-until-clean）

## 二、前向接口：Enabling Specification = 四方对齐的可执行规范

模式原文：产品负责人交付的规格必须证明“重要问题已被提出并已被解决”——否则未探明的需求会在开发中变成昂贵的意外。

这个模式在芯片行业的工业形态已经存在，就是可执行规范：算法、RTL、验证、原型四方对齐同一套接口，用虚拟原型嵌 UVM Scoreboard 做 predictor 交叉验证。“四方都对齐了”就是“重要问题已被提出并解决”的机器可检验版本。

因此 PM 在产品流水线里产出的 Enabling Spec，直接成为工程流水线 L1 意图声明的输入。产品语言到工程语言，不再需要人肉翻译。

## 三、反向接口：工程信号回流，驱动 PM 的三类决策

| 工程流水线信号 | 产品流水线动作 |
|---|---|
| 契约冻结事件(接口/空壳/Golden Ref) | 承诺升级(探索→意向→合同,只认冻结不认压力) |
| 四里程碑门禁通过/未通过 | 发布计划的日期区间收窄或报警 |
| 真实负载画像(Emu/FPGA上采集) | 裁剪决策与场景排序调整 |
| 逃逸率/覆盖率/对齐差异数 | 假设账本更新——每日认知进化的数据来源 |

没有这个反向接口，PM 的判断依据是汇报和感觉；有了它，判断依据是机器采集的工程事实。
