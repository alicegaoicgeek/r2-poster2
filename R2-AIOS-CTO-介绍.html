TECH DEEP DIVE
三个核心技术机制
Thin Harness, Fat Skill — 让知识活过90天
机制一

r2-specx 编译引擎

PDF spec → 结构化JSON → 机器可读 → 驱动交付闭环。

传统方式：人读spec，手写RTL，人工对比差异。
R2方式：spec本身就是编译单元，变更自动触发回归。

机制二

Skill 是 IP，不是便利贴

Skill ≠ md文档。Skill有结构约束、反馈回路、组织背书。

判断标准：删掉这一层剩下什么？
剩下手工流程 = 泥球
剩下裸机 = 真OS

机制三

Debug 波形断言

波形不是"查波形工具"，而是时序行为编码成可自动检查的SVA/assertion。

仿真中自动 PASS/FAIL，不依赖人肉review。
与r2-debug工具链不同维度——Oracle裁判体系。

【演讲备注】这是技术骨干最关心的页面。重点讲"Skill vs md"的判断标准——用户强调过这是核心概念。波形断言要强调"跟r2-debug毫无关系"，Oracle裁判是判决标准，r2-debug是debug工具链。
