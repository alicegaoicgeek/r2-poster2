# R2-小模型执行 + 大模型顾问

Anthropic Extended Thinking · Advisor Routing · 模型间协作新范式

## 01 核心机制：小模型管家 + 大模型顾问

用小模型作为全程执行任务的管家，调用工具、读取结果、迭代推进。当遇到自己判力不够的决策点时，才调用更大的模型来作为顾问。大模型会到达共享上下文，返回一个计划、一个纠错或者一个停止信号。小模型在继续执行——这样的路由方式，效果不错。

小模型 Sonnet / Haiku（执行 · 调用 · 迭代）→ 大模型 Opus（顾问 · 判力不够时）→ 返回 计划 / 纠错 / 停止 → 小模型 继续执行

## 02 评测数据：准确率升 + 成本降

| 组合 | 基准 | 准确率变化 | 成本变化 | 结论 |
|---|---|---|---|---|
| Sonnet + Opus 顾问 | SWE-bench Multilingual | +2.7% | -11.9% | 比 Sonnet 单独跑：准确率升，成本降 |
| Sonnet + Opus 顾问 | Browse.com / Terminal Bench | 超过 Sonnet 单独 | 成本更低 | 同样：加了 Opus 反而更省钱 |
| Haiku + Opus 顾问 | Browse.com | 41.2 分，比 Haiku 单独高 2 倍+ | 成本 -85% | 接近 Sonnet 水平，成本大幅下降 |
| Haiku + Opus 顾问 | 综合 | 比 Sonnet 单独低 29% | 成本 -85% | 简单任务很适用，高吞吐场景有吸引力 |

成本降低原因：Opus 作为顾问让 Sonnet 少走弯路，减少了总 token 消耗。花几分钱请教一次 Opus，省下来 Sonnet 弯路烧掉的可能几毛钱。

## 03 两个重要信号

### 01 API 层面原生支持模型间协作
Anthropic 在 API 层面提供了模型间协作的原生支持。之前需要用户自己写编排逻辑，处理上下文传递、两次 API 调用。现在做一个声明，一次调用中就完成了。上下文 overhead 减少，内容在内部完成。

### 02 定价逻辑变化
Advisor 每次只输出 400~700 个 token 的短计划，按 Opus 的价格算也就几分钱。但这几分钱的指令，让执行的 Sonnet 或 Haiku 少走很多弯路，减少了总 token 消耗。出现加了更高级模型反而总成本更低的现象。

## 04 类比与总结

**类比：专家顾问模式**

找一个专家 24 小时全职工作，成本太高。大部分问题基层同事可以完成，遇到复杂问题时咨询一下专家，只需要按小时支出咨询费用。专家的工作在于指出方向而不是继续执行，所以不需要投入那么多时间。总体单位成本虽高，但总耗量耗金额不会很夸张。
