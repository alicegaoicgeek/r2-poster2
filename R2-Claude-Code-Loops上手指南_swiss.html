01
四类循环 = 四个自动执行档位
TURN → GOAL → TIME → PROACTIVE
01 · TURN-BASED
回合制 · 让它自己验收
把手动验证步骤写成 SKILL.md，让它自己核自己的活
SKILL.md 写清检查项 → agent 每轮自检 → 你只看最终结果
02 · GOAL-BASED
目标制 · 让它自己判断何时收工
确定性标准最有效：测试通过数、分数阈值
/goal 把首页 Lighthouse 提到 90 以上，试 5 次就停
03 · TIME-BASED
时间制 · 让它自己掐点开工
按节奏到来的工作交给间隔循环，不必守着
/loop 5m 检查 PR，处理 review，修掉挂了的 CI
04 · PROACTIVE
主动式 · 整件事全程不用你管
schedule + goal + workflow + auto mode 组合编排
/schedule 跑 routine + /goal 定终点 + workflow + auto 免审批
AUTONOMY
L1你写验收
L2它判收工
L3它掐点开
L4全程自驱
YOU ↓ · AI ↑
02
/goal · 质量 · token — 三块硬机制
STOP · REVIEW · COST
02 · WHY /GOAL
想停时有评估模型挡着
  • 每次想停 → 评估模型检查条件
  • 不满足 → 送回去继续干
  • 确定性标准最有效
  • 测试通过数 / 分数阈值
  • 不用再猜什么算「足够好」
03 · QUALITY
独立 agent 做 code review
  • 新鲜上下文 → 偏见更少
  • 不受干活 agent 推理影响
  • 结果不达标：别只修这一个
  • 把标准编码进 skills
  • 让以后每一轮都受益
04 · TOKEN 省法
六条边界里最易被忽视
  • routine 别跑得比需要更频
  • 间隔匹配变化频率
  • 确定性工作用脚本跑
  • 比模型每次重推便宜
  • /usage 随时看消耗
05
上手三问 — 哪一问答得上来，就把那一块交出去
START SIMPLE
Q1
验证检查写得出来吗？
能 → 交 turn-based
SKILL.md 写清验收步骤
Q2
目标足够清晰吗？
能 → 交 goal-based
数字阈值 / 通过数当停条件
Q3
工作是按节奏到来的吗？
能 → 交 time-based
/loop 匹配变化频率
