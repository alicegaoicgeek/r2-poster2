# R2-用 Claude Fable 5 写代码的核心方法论

地图不是疆域——工作质量的瓶颈在于你澄清"未知"的能力

## 核心框架：四种未知矩阵

| | |
|---|---|
| **QUADRANT I 已知的已知** prompt 里写了什么，你明确告诉 agent 的要什么 | **QUADRANT II 已知的未知** 没想清楚，但至少知道自己没想清楚 |
| **QUADRANT III 未知的已知** 太显然没写进 prompt，但看到了就能认出来 | **QUADRANT IV 未知的未知** 压根没考虑过，不知道不知道什么 |

减少和规划你的未知，就是 agentic coding 的核心技能

## 模型分工：Fable 5 × DeepSeek 组合策略

### Claude Fable 5
顶配模型 · 一头一尾

- 品味、判断、建流程——这些"人的活儿"可以让 Fable 做
- 复盘：把成功经验抽成工作流
- 把失败经验抽成错题本
- 这个钱值得花

### DeepSeek
流程执行者 · 刷流程

- 三个定点做完、工作流建好、错题本建好之后——刷流程
- 干活的活儿好使
- 做决策太平庸

这个组合，目前看来比较合理

"Fable 是第一个让我觉得，工作质量的瓶颈其实在于我自己澄清'未知'的能力的模型"

## 八大实操技巧（按工作流顺序）

| # | 技巧 | 说明 |
|---|------|------|
| 01 | 盲区扫描 | 陌生领域开工前，让 Claude 帮你找到"未知的未知" |
| 02 | 先做原型 | 面对"未知的已知"，用低成本原型试探 |
| 03 | 让它面试你 | 模糊处让 Claude 反过来问你，优先问"会改变架构"的问题 |
| 04 | 给它看参考 | 说不清时给源代码，比截图和描述强十倍 |
| 05 | 先出方案 | 写代码前先出实现方案审阅，重点看数据模型/类型接口/UX |
| 06 | 边做边记 | 维护 implementation-notes.md，记录偏离和边界情况 |
| 07 | 交付说明 | 打包原型 + spec + 笔记 + demo，加速审批和认同 |
| 08 | 验收考试 | 让 Claude 出测验题，全对才合并代码 |
