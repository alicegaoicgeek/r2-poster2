R2-先把课放回正确的位置

上一课把功能签出，这一课在已签出版本上做 PPA：把时序和面积推向更好，并且每一步可追问、可交接、可回放。
上下游关系
签出在前，PPA loop 在后

上下游关系写死——签出在前，PPA loop 在后；功能门禁是前提，本课主线是优化 loop 怎么转。

设计对象
控制微序列器

约四千行 RTL、约一万六触发器，含预取、译码、标志更新及跨域控制耦合；功能点与骨架、P0 门禁已在上一课签出，本课输入为稳定版本，以及写入 GOAL 并锁定的 WNS/TNS/Area 基线。

深水区与 LOOP 边界

它不是外设型 IP，而是控制耦合型设计，深水区在关键路径锥、控制扇出、晚选择与长组合结构——PPA 主 loop 只在这些点上诊断根因、按三三制加载策略出候选，经综合与对拍过滤后，相对固定基线算 score 并决定提升或换挡；不重做功能定义，不移动基线，门禁失败结果不进评分。缺稳定版本或基线三数，禁止启动 loop；没有原点，快慢对比无效。

进入检查清单
每一项都必须可验证

进入本课的检查可以做成清单，但每一项都必须可验证：

  • 稳定版本路径存在
  • 上一课签出证据可定位
  • GOAL 中基线三数与综合报告一致
  • INDEX 可追加写入
  • 策略库可按名称调用并留痕
  • 门禁脚本可重复执行

任一项只靠口头确认，不算通过。

SCORE 比较规则
门禁通过集合内比较

score 比较只在门禁通过集合内进行。三个候选若只有一个通过，就在这一个与当前最优之间比；若全部失败，本圈无 score 决策，只产出失败模式与诊断复审标记。

禁止在失败集合里挑"看起来 WNS 更好"的版本晋升——那是最常见的退出 loop 方式。

R2-PPA 先建成 loop，否则只是改 RTL

PPA 要成工程，必须建成一条可重复执行的 loop，并且每一轮 loop 在 INDEX 留下可检索记录。
LOOP 七步 · 顺序固定
  1. 01读 INDEX，接上失败策略与当前最优
  2. 02诊断 critical path，把慢写成根因——例如组合逻辑深度过大、控制信号高扇出、FSM 输出拖长下游、晚选择或宽比较——而不是只写"路径太长"
  3. 03按根因做三三制，并行三个不同策略家族候选
  4. 04综合后跑对拍或 equiv_check，失败直接丢、原因入库
  5. 05仅对通过者相对进入本课时锁定的 WNS/TNS/Area 基线算 score，更优则提升并清零无改善计数，否则计数加一
  6. 06写回 INDEX 后再开下一圈
  7. 07计数触达阈值就换家族或回退切分边界；达到版本上限或 GOAL 目标就停表，交人最终签核
分工不能混
四种角色各司其职
角色职责
模型提出改写
工具给出时序、面积与等价结果
INDEX保存根因、策略、门禁结论与 score 轨迹
人只在灰区抉择并在结束时签字

跳过门禁谈快慢，或移动基线制造假进步，都视为退出 loop。

跑通 PPA 的判据
四个"能不能"
  • 固定基线还在不在
  • 失败能不能检索
  • 提升能不能回放
  • 无改善会不会强制换挡

满足这些，优化才是 closed loop；不满足，就仍是改 RTL 的冲动。

R2-三三制：同一根因并行三个不同策略家族

同一根因并行三个不同策略家族（扇出、切割、重构），统一综合与门禁，按 score 排序综合，形成下一圈问题再开三路。
根因 → 策略家族对应表
根因类型优先策略家族说明
扇出类寄存器复制或局部缓冲降低扇出驱动的路径延迟
组合过深切割与分段寄存打断长组合链，插入流水
晚选择选择前移或晚绑定提前选择信号或延后绑定
长表达式拆中间量与公共子表达式提取公共项，减少重复计算

对应关系不是教条，但必须写进当圈 INDEX，换挡时才知道试过哪一科。同家族参数变体不计入三路。

两组护栏
目标只有一个

目标只有一个：相对锁定基线的 score 改进。

  • 护栏一：门禁失败不得评分
  • 护栏二：基线不得移动，失败必须入库

生成可以快，错误不能漏。

对着当前根因把本圈做透，提升或确认失败后立刻复盘，把有效手法写入 INDEX/Skill；拖到事后，沉淀质量会明显下降。策略家族与根因的对应要事先可查，避免临场拍脑袋。

三三制执行要点

三三制在 loop 里这样用：同一根因并行三个不同策略家族（扇出、切割、重构），统一综合与门禁，按 score 排序综合，形成下一圈问题再开三路。门禁与 score 能自动则自动；灰区才人工，口径事先写进 GOAL。

R2-什么叫改进：固定原点上的可复算比较

比较必须锚定进入本课时锁定的 baseline，而不是锚定"上一圈我最喜欢的结果"。
固定原点
基线锁定，禁止改写

过滤之后才有比较，比较必须锚定进入本课时锁定的 baseline，而不是锚定"上一圈我最喜欢的结果"，否则分数会在移动靶上自我庆祝，团队也会在自我庆祝里失去对真实进展的感觉。

尺在进 loop 时钉死：基线 WNS、TNS、Area 来自第一版签出综合，之后禁止改写。

改进的唯一合法含义
先过门禁，再在固定原点上得更好 score

先过门禁，再在固定原点上得到更好的 score，然后才配提升为新的 major 版本。看起来更快却过不了等价，不是改进；只对内轮自嗨、却说不清相对签出点如何，也不是改进。

三三制给出的多个候选，在 PPA loop 里并不直接"谁看起来快谁上"，而是先过等价/对拍，再在同一把尺子上比。

SCORE 公式
比较量定义为越小越好的 score
分项归一方式系数说明
WNS基线减当前、再除以基线绝对值0.5最坏路径时序
TNS同上0.35整体违例
Area相对基线的增量比例0.15面积变化

计算时符号方向必须事先写死——改善对应更低 score——避免在 WNS 为负的习惯下把方向算反。面积相对基线恶化超过 10% 时，额外加 0.5 的惩罚项，否则惩罚为 0；基线绝对值过小则该分项按 0 处理，避免除零。

示意计算

基线 WNS −0.50、候选 −0.20，时序归一为 −0.60；若 TNS 归一 −0.40、面积归一 0.05，加权后约 −0.43。

面积一旦跨过 10% 阈值，惩罚会抵消部分时序收益，避免用面积换数字。

可重放性
loop 与手艺的差别

loop与手艺的差别在于能否重放：谁被提升、谁被丢弃、谁因面积惩罚被压下，都应能在 INDEX 里回放。

尺固定，争论才能回到数上；尺随情绪移动，经验只能变成传说，不能变成原则。

R2-运行时：继续、换挡、停车与可见性

loop一旦转动，管理问题往往比生成问题更致命：生成可重来，管理失灵会污染记忆链。
STATE 01
继续

未达版本上限、未达目标、未触发切换，就继续。候选过门禁但未超过当前最优，无改善计数加一。

STATE 02
换挡

计数触达阈值，换策略家族或回退切分边界。禁止同一家族靠加试次冒充进展。

STATE 03
回退

系统性回归，先回上一稳定 major。回退时从该版的计数与最优指针恢复，不凭记忆口头回退。

STATE 04
停车

版本上限、整圈无提升、GOAL 达成。停表后必须人签核。日常转 loop 可以自动，停表不能无人负责。

失败率陡增，先查诊断与改写幅度，不先放宽门禁；家族集合不增长而轮次空涨，说明三三制名存实亡；计数到顶仍不换挡，说明管理失灵。没有切换条件的搜索会在局部最优自我说服；没有失败分类会把"功能改坏"与"方向试尽"混为一谈。

异常处理 · 三类
异常处理
三候选全失败回到诊断并缩小改写幅度，不降门禁
频繁触面积惩罚检查是否用面积换时序并换家族
INDEX 与 RTL 不一致以可复现门禁结果为准，先停止 loop 修记录

宁可停止 loop，不污记忆。

版本管理
上限与计数
  • 版本上限默认到 v10，或 GOAL 另写
  • 一个 major 轮次无提升即可停
  • 工程上可并行使用连续多圈无改善强制换挡
  • 两者都要在 GOAL 声明
  • 提升 major 时无改善计数归零
  • 回退到稳定版时，从该版计数与最优指针恢复

避免执行到一半争论停不停。

R2-沉淀：Loop 真正留下的资产 · 第一圈怎么跑通

失败不入库，资产为零。数字是收成，记录是土壤。第一圈只追求闭合。
沉淀 · 第六章
INDEX 每圈至少增加
  • 根因
  • 三策略名
  • 门禁结果与失败原因
  • score
  • 决策
  • 无改善计数

多圈之后，还应有相对 baseline 的 score 轨迹，以及"某策略家族在某类根因上无效"的负样本地图。控制微序列器路径密集，人脑记不住禁忌；策略名会引诱重复同一种错误。失败不入库，资产为零。

考核与沉淀
数字是收成，记录是土壤

考核若只奖数字、不奖失败入库，loop 会幻觉：挪原点、藏失败。数字是收成，记录是土壤。禁忌可检索、最优可回滚、争论能指回证据，才是复利。PPA 里稀缺的往往不是更强生成，而是失败可检索；失败显形，学费才降下来。

有效手法在提升或确认失败后立刻写入 INDEX/Skill，不要拖到事后。Skill 沉淀的是"某根因 → 某家族 → 门禁与 score 结果"，不是空口号。

第一圈怎么跑通 · 第七章
第一圈只追求闭合
  1. 01核对稳定版本与基线三数，写入 GOAL（含公式符号与版本上限）
  2. 02建立 INDEX v0，当前最优=签出版本，无改善计数=0
  3. 03跑 timing，写合格根因
  4. 04按根因开三路候选
  5. 05同一门禁脚本跑完
  6. 06对 PASS 算 score
  7. 07决定提升或计数
  8. 08整行写入 INDEX

结束时团队必须能指着记录说：试了什么、打死了什么、相对原点如何、下一圈禁止什么。指不出来，loop 未接通。第一圈通了，才有资格谈百分比；第一圈不通，百分比只是噪音。

FAIL 01

根因写太粗

FAIL 02

三路实为一路

FAIL 03

门禁脚本中途更换

FAIL 04

基线被改成上一候选

这四类在启动 loop 检查里就要挡掉：根因必须落到可匹配家族；三路家族名不同；门禁命令与报告路径写死在 GOAL；基线三数只写一次。从微序列器这类控制核的经验看，PPA loop 前几圈的价值多半不在大涨百分比，而在快速排除无效家族——排除本身就是进展，前提是排除被写进 INDEX。

第一圈到第三圈的管理重点是：诊断是否越写越具体，三路是否真不同，失败是否可分类，基线是否仍固定。
前几圈管理重点
把 loop 的卫生做干净
  • 诊断是否越写越具体
  • 三路是否真不同
  • 失败是否可分类
  • 基线是否仍固定

百分比是后几圈的话题；前几圈把 loop 的卫生做干净，后面的百分比才有讨论资格。

第二圈起每次先读 INDEX，加载负样本与最优指针，再诊断与三路取样。评估始终对同一基线，不把基线挪到"上一圈最优"。圈数增加后，看 score 轨迹与无改善计数，比开会争论"再试试"干净。

迁移管法
换控制型 IP 时
  • 固定原点
  • 先工具后分数
  • 三路取样
  • 失败入库
  • 计数换挡
  • 人签停表

不迁移某次 WNS 故事。功能签出是上车；能走多远取决于优化是否在约束里转、是否把记忆留下。

完成判据 · 五条同时满足
  1. 01合法停表条件触发
  2. 02最终版本门禁通过
  3. 03INDEX 自 v0 连续可审
  4. 04有相对固定基线的对比说明
  5. 05人已最终签核

缺一不可。

GOAL · 启动前至少写清六项
  1. 01等价优先于 PPA
  2. 02锁定的 WNS/TNS/Area 基线三数
  3. 03score 公式与符号约定
  4. 04版本上限
  5. 05门禁命令与报告路径
  6. 06停表与换挡条件

缺基线或缺公式，禁止进入 major v1。

人签核要求

最终放行前抽查至少一条 FAIL 记录是否有原因、一次提升是否可按 INDEX 复算 score、基线三数是否与 GOAL 一致。抽查不过，不签字。签字文本需确认门禁通过、INDEX 连续、对比相对固定基线、灰区已处理。

本课承诺

本课不承诺固定小时内的固定改善百分比。承诺的是：在功能已签出的前提下，你有一条能转、能记、能换挡、能停表的 PPA loop，并且第一圈起就可以按记录工作，而不是按聊天记录猜。

让模型提出可能，让工具拒绝幻觉，让 INDEX 积累禁忌与最优，让人在该负责处签字。loop 转起来，PPA 才开始；loop 管起来，PPA 才结束得了。
