# 1 · 2 认知框架 分析拆解

R2 - VPlan 五层方法论 · 第一二层

把坐标系从TB移到DUT。围绕DUT组织，先说清楚不验什么。然后四步从规格走到Feature矩阵。

## 第一层 · 认知框架 — DUT中心四准则 + 使命声明四要素

### 01 围绕DUT组织
按功能分Feature不按TB组件。高层决策不写实现。描述场景不列测试名单。

- **RTL** 硬件投影 时钟边沿行为：时序、流水线、握手 — 仿真+Formal+Emu 的输入
- **OOM** 程序员投影 寄存器接口语义、操作模型 — Golden Model, QEMU+Lockstep 的参考
- **DOM** 配置投影 验证环境的声明式配置空间。形式化表达后注入四个维度：Formal侧property参数、仿真侧uvm_config_db、Emu侧C测试参数、QEMU侧固件配置

### 02 先说清楚不验什么
验证目标·应用场景·质量指标·明确排除项。范围锁定后不可争论。

## 第二层 · 分析拆解 — 四步从规格走到Feature矩阵

1. **选一个维度切进去**  
   按规格/Block/Feature/用例/风险/场景——每次只用一种，隔离完成后独立描述分析

2. **写 DITL（芯片的一天）**  
   上电→启动→数据流→中断→异常→关闭。侦探视角发现规格未写明的隐含行为，压缩到10–15条mini-DITL。

3. **分四类场景 × 六个维度**  
   独立/混合/合法异常/非法 — 每类场景乘六维度 = 24格可审计矩阵。若某场景无法归类，说明隔离不完整。

4. **六问法逐条审**  
   验什么 · 含哪些功能 · 为何这个粒度 · 怎么激励 · 怎么检查 · 属哪层哪阶段。每个空白格是已知风险。

## 六个维度 · 随 IP 类型变化（以算法类 vs 接口与互联类为例）

| 维度 | 算法类 (ISP/Codec) | 接口与互联类 (PCIe/NoC) | 维度如何变 |
|------|-------------------|------------------------|-----------|
| 总线类型 | 内部流水线，对外标准AMBAPCIE/AXI/ACE/CHI 多协议混用 | 单总线→多协议异构 | |
| 事务类型 | Mem进Mem出，数据块处理 TLP/DLLP格式 + 多Master 事务模式 | 数据包→协议事务 | |
| 数据流路径 | 固定处理管线，输入→处理→输出 | TX/RX双通道 + NxM多Slave路径 | 固定管线→拓扑路由 |
| 目标Block | 内部计算单元 (MAC/ALU) | RootPort/Endpoint + 各Slave独立握手 | 内部单元→外部IP |
| 配置空间 | 算法参数（滤波器系数/精度） | LTSSM状态机+链路参数+QoS仲裁 | 算法参数→协议策略 |
| 时序条件 | 吞吐率约束，流水线气泡 | 链路训练+恢复+多Master竞争 | 计算节拍→动态冲突 |
