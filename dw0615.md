# Dynamic Workflows 知识卡

## 一句话
别人以为它是"并行跑几百个 agent"；真相是它让每一步工作都发生在全新、干净的 context 里。

## 本质
真正的货币是 context 质量，不是 agent 数量。LLM 只在 context 前段做得最好（Opus 4.7 过 200k 即 unusable；afro88："first 20% sweet spot" "clean context windows"）。workflow 的唯一本职：保证没有一步工作发生在被污染的 context 里。并行是副产品。

## 核心理念
不要问"模型能不能更聪明、记住更多"，要问"怎么让每一步都在 fresh context 里发生"。把模型做大反而更糟：context 越大、填得越满，退化越快。

## 【context 流向 · 新旧对照】（这才是跃迁所在）

| | 旧：subagent (Claude 当 orchestrator) | 新：workflow (脚本当 orchestrator) |
|---|---|---|
| 中间结果 | 每个都回灌 Claude context | 留在脚本变量，永不进 LLM |
| Claude 只见 | 全过程逐轮 | 仅最终答案 |
| 可 fan out | 一小把（orchestrator 先饱和） | +1~2 个数量级（bcherny） |
| 计划载体 | Claude 逐轮判断 | 脚本持有循环/分支/中间结果 |

## 【硬参数】
- ≤16 并发
- ≤1000 agent/run
- 同会话断点续跑
- subagent 跑 acceptEdits 继承 allowlist
- 脚本无 fs/shell
- 无运行中干预（仅 agent 权限申请可暂停）
- 需 v2.1.154+

## 【质量机制】
人类无法 review 100x 产出 → 验证必须由 agent 承担：多 agent 独立求解 → 对抗 agent 抢在交付前证伪 → 迭代到收敛，达到单次遍历到不了的质量。

## 【四种用法】
1. **关键词 workflow**：全库审计 / 数百文件迁移 / 需多角度 stress-test 的方案
2. **/effort ultracode**：xhigh，Claude 自判是否要 workflow，可串联多个（理解→改→验）；重活后切回 /effort high
3. **/deep-research <问题>**：fan out 多源搜索 → 交叉验证 → 逐条投票 → 带引用报告，未过验证的 claim 被剔除
4. **对抗性验证**：错误代价高的活：独立尝试 + 对抗 agent 打破结论后再交付

## 【战绩 · 全是真实交付】
- Bun Zig→Rust：75万行 / 11天 / 99.8% 测试通过
  ① 映射每个 struct 的 Rust lifetime
  ② 数百 agent 并行逐文件移植，每文件 2 reviewer
  ③ fix loop 跑通 build+test
  ④ overnight 清冗余拷贝逐处开 PR
- Anthropic 内部 token −15% · WASM/原生模块→TS，CPU/内存 ×2–10 · 权限误报 −45% · Agent SDK 启动 −61% · 69 个 PR 删 1万+行
- 基准 SWE-Bench Pro 69.2%（GPT-5.5 58.6% / Gemini 3.1 Pro 54.2%）· agentic compute 83.4%
- 企业 Klarna 跨微服务安全审计（缺失认证/脆弱校验/复制粘贴漏洞）· CyberAgent 生产服务对抗性加固
