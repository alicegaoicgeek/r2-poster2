# R2-CLAUDE.md 写太多，反拖累 Agent

Anthropic 上下文工程 + ETH 研究 · 方法

## 核心判断

给 Agent 的上下文不是越多越好。  
上下文是有限的注意力预算。  
Token 越多，召回越差；规则越满，执行越钝。

**ETH 2026-02:**
- 人写 context: 约 +4%
- AI 生成 context: 反而 -3%

问题不是没有规则。  
问题是把低信号规则也塞进系统提示。

## 三条洞察

- 注意力预算管理，不是知识总量管理
- 子代理各自只回 1000–2000 token
- 目标是最小高信号 token 集

## 上下文该怎么写

**该做:**
- 维护高信号最小集
- 定期删除冗余
- 压缩 + 笔记 + 子代理

**不该:**
- 规则越写越多
- 只增不删
- 盲信 AI 生成 context

## 塞满 vs 少而精

**塞满:**
全写进去 → 越堆越多 → 关键规则被淹没  
低信号堆积 → 关键高信号被压  
→ 召回差 / 执行钝

**少而精:**
高信号最小集 → 按需读取 → 召回更准  
高信号最小集 → JIT 检索 → Agent  
→ 召回差? / 执行快

## 官方方法论 5 条

1. **系统提示适度:** 不过度，也不空泛
2. **工具别堆:** 功能重叠越多，模型越选不对
3. **JIT 检索:** 文件按需加载，别全灌
4. **Compaction:** 近上限就压，留决策点冗余
5. **笔记式记忆:** 写 NOTES.md，需要时读回
