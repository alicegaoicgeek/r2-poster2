每轮结束后跑 npm test --coverage 并检查报告

必须有新测试文件或修改

无新 failing test

时间制

Time-based · 每日代码健康检查

/loop 1d · 22:00

START PROMPT

/loop 每天 22:00
检查所有未解决 issue + PR
对每个高优先级 issue 自动生成修复建议并创建 draft PR
停止条件：所有优先级 issue 处理完毕或超过10个

SKILL.MD 关键

优先级：label=bug + assignee=me

修复建议必须含代码 diff

+ 测试命令

CHIP · HARD TASK · ACCEPTANCE CONTRACT · MULTI-ROUTE · ADVERSARIAL

R2- 难任务合同式工作法

把「请帮我做好 X」改成「按合同交付可验收的 X」——验收钉死 · 伪完成拉黑 · 多路线探索 · 独立找茬

ANCHOR

难任务不写固定 SOP，而写 可验收合同 ：只钉完成标准 · 定义与边界一次说清 · 写明什么不算答案 · 动态多路线 + 生成与审查分离

01

不写 SOP，只钉「什么叫做完」
ACCEPT

机制： 路径未知时步骤常错；认真执行错误路线 → 漂亮废稿

RTL： 不写「先仿真再综合」→ F1–Fn 全绿 + 无 X + 违例≤0

验证： 交付 = 可复现 pass / 覆盖率阈值 / 断言列表

评审： 完成 = 风险清单关闭态，不是「讲过一轮」

02

定义 · 范围 · 边界一次说清
SCOPE

机制： 多数失败是任务理解漂移，不是推理力不够

规格块： 是什么/不是什么；允许 stub？复位语义？

禁止假设： 只证特例、只跑 demo 规模、默认连通

空情况： 空输入、全 0、无事务；长任务重复钉目标

03

白名单 + 黑名单：什么不算答案
NEGATIVE SPEC

| 场景 | 完成标准（示例） | 一律不算完成

| RTL / 模块 | 失败测试先红再绿；关键断言可杀错实现 | 只有 happy path；stub 能骗过断言；无边界用例

| 验证计划 | 覆盖点可映射到 test；退出条件可测 | 「测了主要场景」无清单；固定规模跑完即完

| 疑难 debug | 根因 + 最小复现 + 防回归检查 | 现象描述充数；换配置碰运气；无证据链路

| 设计文档 | 接口 / 时序 / 错误处理可据此实现 | 分类标签无机制；把未决问题当 routine

| 形式 / 证明 | 结论精确覆盖陈述；边界与反例处理齐全 | 只证特殊子类；归约到另一未证命题；循环定义

| 综合 / 时序 | 约束下可实现 + 违例归零或可解释 waiver | 「先看一下」无报告；隐瞒违例；无对应约束源

04

动态多路线 + 独立对抗审查
2–6 AGENTS

Portfolio： 3 条互不兼容路线（结构 / 代数·形式 / 反例压力）

独立早期： 勿告知「当前最看好」，防集体收敛同错

Registry： 按机制思想分组，不按措辞；同族挤堆则分流

Blocked： 卡在与原题同难的引理 → 停烧，除非新机制

Adversarial： 另会话只拆：偷换定义、漏边界、循环论证

回传： 只收引理 / 构造 / 波形 / 反例；拒收「有进展」

05

日常可贴的「工作合同」壳
TEMPLATE

任务 一句话目标

定义 关键名词 3–7 个 + 空情况

完成 必须同时满足的可跑验收

不算 伪完成黑名单（特殊子类 / 归约未证 / 固定规模）

探索 最多 N 条独立路线；blocked 标记；禁 routine 空话

交付 过 audit 才返回；否则：最强已证 + 精确缺口

审查 另开角色只找漏洞，不提新方案

06

本周最小实验 · 何时不必全套 · 芯片日常映射

LIGHT 极赶时间 → 至少保留「不算完成」三条

SKIP 路径已知且验收单一 → 短指令即可

模块交付 功能点清单 + 仿真绿 + 边界用例

覆盖收敛 缺口映射 test，禁「差不多了」

疑难 bug 3 路线假设并行，独立证伪

设计评审 风险关闭表 + 审查人只拆不提新案

把难任务写成 可验收、可审查、可纠错 的合同——不规定唯一道路，但把终点、边界、失败条件与审查机制写死。

# 幕3 · 11类全景：编排断裂的化石

> 素材处理：本幕使用素材5条，其中逐字引用2条，改写引用3条。

---

**11类不是分类学。是编排断裂的化石。**

这不是一个目录。这是一个伤口愈合后留下的疤痕形状。

你问为什么是11类，不是9类、不是15类、不是200类？

Shannon信道容量给出了答案。

20个Skill的时候，分类准确率还能维持在90%。超过这个数，分类本身就开始产生噪声。200个Skill的时候，准确率跌到20%——比抛硬币好不了多少。

这不是因为模型变笨了。是因为**分类维度超过了信道的承载能力**。

11，是信道约束下的最优收敛。不是拍脑袋选的数字。是减法运算的幸存者。

---

**心脏不是最重要的器官。**

这是芯片设计里最反直觉的一句话。

specx，R2的11类Skill之一，是所有六条产线的同源真理源。它不是"最重要"的——它是最先停的。

FP55，一个并发写缓冲的死锁bug。177条FP规则、1130条断言、8美元算力、25分钟，首次运行即捕获APB FSM死锁。100%功能覆盖率，97.7% Toggle率。

这个bug不是人工注入的。是完全由Spec推导的断言，在TDD流程中自动捕获的。

它证明了什么？证明了**翻译器不可信没关系，验证器可信就够了**。specx停了，所有器官都停。因为它不是心脏——它是心跳本身。

---

**类别从断裂中长出来。**

Boris，Claude Code的作者，说过一个频率原则：

> 创建信号是频率，不是重要性。

Loop沉淀为Skill，Skill沉淀为Routine。这不是一个设计过程，是一个**演化过程**。

编排先有调用需求，Skill类别随后形成。不是先画分类表再填内容。是先有断裂，再有类别。

11类，是11个历史断裂点的命名。每一个类别背后，都有一段产线停摆的记忆。

**倒过来就不是这个全景了。**

---

**减法方向：编排逻辑从Skill抽离到Harness。**

这是幕3独有的认知增量。

Skill的数量不会指数增长。未来的方向恰恰相反——编排逻辑从Skill内部抽离，进入Harness。

Skill做减法，Harness做加法。

11类不是终点，是当前信道容量约束下的临时收敛。当Harness的编排能力提升，更多的Skill可以被同一套逻辑组织。

---

**收束**

幕2回答了单个Skill怎么变IP。幕3回答了芯片产线需要多少Skill、为什么是这些、它们怎么长出来的。

下一幕：化石怎么被组织成活着的产线？

> 金句：不是先有分类后有Skill，是先有断裂后有类别。

> 带走物：编排断裂诊断三问
> 1. 你的产线最近一次停摆在哪个环节？
> 2. 那个环节有没有对应的Skill类别？有→补Skill，没有→你在见证一个新类别的诞生。
> 3. 这个类别是否已经在11类里？不在→它就是第12类的候选。

# R2 Poster — 文字转图片工作流

把 Markdown 文字内容转为 R2 Swiss 风格的设计图片。输入一篇文章，输出一张排版精良的 PNG。

## 速查

```
读源内容 → 蒸馏 → 容量估算(🛑超预算裁内容) → 写HTML → Chrome截图 → Playwright验证 → agent QA → 迭代至0溢出
```

**核心迭代循环**（每轮约 30 秒）：
```
写/改 HTML → 截图 → node verify-poster.mjs → 有溢出？→ 按报告收紧间距 → 重跑
                                              → 0溢出？→ agent QA 视觉检查 → 通过？→ 交付
```

## R2 Swiss 设计系统（固定，不可偏离）

### 色彩

| 用途 | 色值 | 说明 |
|------|------|------|
| 背景 | `#F7F4EF` | 浅米白，不可换 |
| 主文字 | `#141414` | 近黑 |
| 次要文字 | `#666666` | 灰色 |
| 三级文字 | `#999999` | 浅灰 |
| 唯一强调色 | `#E8501F` | 橙色，全图只用这一种 accent |
| 分隔线 | `#D5D0C8` / `#E8E3DB` | 灰色规则线 |

**禁止**：蓝、紫、绿、霓虹、渐变、阴影。

### 字体

```css
--mono: "SF Mono", "Menlo", "Consolas", "JetBrains Mono", monospace;
--sans: "PingFang SC", "Microsoft YaHei", "Helvetica Neue", sans-serif;
```

不依赖 Google Fonts。工具名/代码用 mono，正文用 sans。

### 间距系统

基于 8px 网格：8, 16, 24, 32, 40, 48, 56, 64, 72, 80, 88

### 布局原则

1. **网格优先**：所有元素按隐形网格排布，有明确对齐关系
2. **强排版不靠装饰**：视觉张力来自字号、字重、间距、留白
3. **非居中但有秩序**：左对齐、偏置构图、大面积留白
4. **少色高对比**：只用上述色彩系统
5. **信息层级清楚**：一眼标题 → 二眼结构 → 三眼细节
6. **几何化表达**：用线、块、圆、规则表达概念
7. **右下角品牌**：`R2芯片智能体`

## 工作流

### Step 1: 读源内容

读取用户指定的 Markdown 文件或粘贴的文本。理解核心结构：标题是什么、有几个章节、最核心的洞察是什么、有哪些可做成速查表的数据。

### Step 2: 内容蒸馏

公众号贴图的本质是**视觉钩子**——不是全文复制，是提炼骨架。

蒸馏规则：
- **保留**：标题、核心洞察（1-2 句）、分类/章节结构、速查/对比表格
- **删除**：段落论述、举例、展开说明、脚注、社区链接、安装命令
- **合并**：多级标题扁平化，相似内容合并
- **速查表**：原文有对比/选择表的，保留 6-8 行最核心行

每张图**只有一个主信息**，不塞满。

### Step 2.5: 容量估算 🛑（写 HTML 前必做，防溢出）

固定画布 = 固定内容预算。**蒸馏后的内容如果超过预算，继续裁内容，不缩字号。**

**1080×1440 画布容量模型**（80px 左右边距，48px 上 / 44px 下 padding，正文 24px 基准）：

| 区域 | 字号 | 行高 | 904px 宽每行容纳 | 硬上限 |
|------|------|------|-----------------|--------|
| 核心数字 | 260-280px | 0.84 | — | 1 行 |
| 标题 | 48-56px | 1.08 | ~14 中文字 | 1 行 |
| 洞察/副标题 | 24px | 1.45 | ~28 中文字 | **≤2 行** |
| 分类/章节项 | 28px | — | ~25 中文字 | **≤7 项** |
| 每项描述 | 24px | 1.4 | ~28 中文字 | **≤1 行/项** |
| 速查表行 | 24px | 1.4 | — | **≤5 行** |
| 分隔间距总和 | — | — | — | ≈500px（各 section 间 gaps） |

**估算公式（24px 正文版）**：

```
总高度预算 = 1440 - 92(上下padding) = 1348px
固定消耗 ≈ 标签40 + 数字218 + 标题52 + 洞察70 + 规则线35 + gaps≈400 = 815px
可变消耗 = 分类项区 + 表格区
分类区 ≈ 项数 × 100px（名称28px + 描述24px 单行 + padding）
表格区 ≈ 行数 × 40px + 表头30px
可变消耗 + 固定消耗 ≤ 1348px ✓ 否则裁内容
```

**三项硬约束（写 HTML 前必须满足，24px 正文基准）**：
1. 分类/章节项 ≤ 7 个
2. 速查表行 ≤ 5 行
3. 洞察文字 ≤ 56 个中文字（2 行 × 28 字）
4. 每项描述 ≤ 28 字（1 行），超过则精简文案不换行

### Step 3: 选定尺寸

| 场景 | 尺寸 | 用法 |
|------|------|------|
| 公众号贴图 | 1080 × 1440 (3:4) | 文章内图片，竖版信息图 |

#### CSS 防御布局规则（防溢出三板斧）

```css
/* 规则 1：body 硬边界 */
body {
  width: 1080px; height: 1440px;
  overflow: hidden;           /* 任何溢出直接裁掉 */
}

/* 规则 2：flex 子元素不撑破容器 */
.poster > * {
  flex-shrink: 0;             /* 大多数 section 不压缩 */
  min-height: 0;              /* 关键：允许 flex 子元素小于内容高度 */
}

/* 规则 3：文本容器有 max-height 兜底 */
.cat-detail {
  max-height: 40px;           /* 2 行 × 12px × 1.55 ≈ 37px，四舍五入 */
  overflow: hidden;           /* 超长工具名被截断而非溢出 */
}

/* 规则 4：表格不撑破 */
.ref-table {
  table-layout: fixed;        /* 列宽严格按 th 声明，不受内容影响 */
}

/* 规则 5：长单词/长代码强制换行 */
.cat-detail, .ref-table td {
  word-break: break-all;      /* 超长工具名/URL 强制断行 */
  overflow-wrap: break-word;
}
```

**为什么这些规则重要**：
- `min-height: 0` 是 flexbox 最常见的溢出 bug 来源——flex 子元素默认 `min-height: auto`，会拒绝缩小到内容高度以下
- `overflow: hidden` 是最后防线——如果前面所有约束都失效，文字被截断比整个布局崩溃更好
- `table-layout: fixed` 防止表格内容撑宽列 → 挤压相邻列 → 文字叠加

## 自动布局原理

r2-poster 的布局策略是「半自动」——不是纯算法生成，而是**约束模板 + 内容自适应**。

### 为什么不做全自动布局

全自动布局（内容丢进去自动排版）在固定画布上不可靠，根因是：

1. **文本换行不可预测**：同样的字数，中文/英文/代码混合时每行实际宽度完全不同。CSS 渲染后才能知道真实行数。
2. **内容结构差异大**：有些文章 3 个章节，有些 12 个。统一模板要么太空要么溢出。
3. **设计感来自「人做决策」**：字号的跳跃比例、橙色用在哪、哪里加规则线——这些是设计判断，算法做不了。

### 当前策略：三层约束

```
Layer 1: 内容预算（事前预防）
  └→ 蒸馏后内容超过预算 → 裁内容，不缩字号

Layer 2: CSS 防御（渲染时兜底）
  └→ overflow:hidden + flex-shrink + max-height + table-layout:fixed

Layer 3: 截图质检（事后验证）
  └→ sips 验证尺寸 + 肉眼检查 + Chrome evaluate 查溢出
```

### 如果确实需要更自动化的方案

两个方向可以探索：

**方向 A · 多页分流**：内容太多时不分页压缩，而是自动拆成多张图（幻灯片模式）。每张图独立完整，翻页查看。这比硬塞进一张图更合理。

**方向 B · 两 pass 渲染**：第一 pass 用宽松字号渲染 → Playwright 量出各元素真实高度 → 第二 pass 按实际高度动态调字号/间距。成本是渲染时间翻倍，但能处理内容量差异大的场景。

当前默认用三层约束策略，多页分流在内容明显超预算时手动触发。

# 驾驭工程：榫卯结构的因果同构论证

## §01 地震来了

AI coding最大的坑不是AI不会写代码。
而是你没有告诉他什么不能写、什么不能改、什么不能碰。
AI会写代码，但它不懂边界。
AI会完成任务，但它不懂代价。
AI会优化逻辑，但它不懂世故。

在AI眼里，废弃代码就是垃圾。但在工程里，废弃代码可能是历史兼容。
在AI眼里，几个状态只是字符串。但在工程里，那是前端、后端、任务流、数据库共同遵守的契约。
真正贵的不再是写代码，是管理Agent Loop。

最夸张的一次，只是让Claude修一个前端显示问题，结果它顺手改了十几个文件，DTO改了，状态机动了，兼容逻辑删了，返回结构也变了。Bug是修好了，但项目差点崩了。
Uber的工程团队给工程师开通了AI编码工具，四个月后烧掉了全年的AI预算。不是模型太贵，是没有人在每个loop上设MAX_ITER、Token上限、金额预算。
张三用Agent调通了一个偶发失败的测试，花了4小时试了5条错路，最后发现是时钟域crossing的边界条件没处理好。李四第二天遇到了几乎一样的问题——从零开始。张三的经验留在他的私人聊天记录里，对李四没有任何帮助。

然而，应县木塔近千年经历40多次大地震不倒，不是因为它比钢结构更硬——恰恰相反，是因为它"让"了。
地震来了结构可以晃动、滑移，把冲击能量变成摩擦热散掉。
它的秘密是榫卯的间隙设计和负反馈自锁。
有缝不是做工粗糙，是设计出来的。
不是死扛，是让。让不是放纵——让是有设计的退让，在退让中把外部压力变成内部连接力。
AI越强，项目越崩，正是因为"你没有告诉它什么不能碰"。这正是卯口缺失的后果。

这种"让"的智慧，如何在工程中找到对应的连接范式？问题的答案，藏在三种根本不同的连接方式之中。

> 出处：566Harness Engineering驾驭工程-AI Coding四层治理.md、861-榫卯结构 harness+loop工程.md、689-主题5-循环工程与持续进化.md、528!!!!r2-aios.md

## §02 三种位置

AI工程有三种连接范式，它们的区别不在表面，而在力的传递方式上。

第一种是钉子。Harness不是钉子。钉子靠的是贯穿——硬打进去，靠摩擦力死撑。钉子结构的系统是硬编码规则、if-else流程、固定脚本。Prompt Engineering时代写的"万能模板"就是钉子——一次性钉死，不给退让空间，久了就松。

第二种是焊缝。Harness也不是焊缝。焊缝靠的是熔为一体——微调模型权重，永久粘合，不可拆。Fine-tune模型就是焊缝——强是强，但拆不了，外部压力一来整体变形。

第三种才是榫卯。驾驭工程（Harness Engineering）就是设计卯口形状的工程学——决定边界在哪、空间多大、壁在哪里。榫卯的精妙不在缝，在缝让榫头挤上卯口壁的那一刻。大力来时，榫头越往里顶，接触面越大。是燕尾榫——头大身小——越压越把卯口两边撑开。木头被撑变形后想弹回去，回弹力反过来把榫头死死夹住。越压越紧，天生防松。

应县木塔近千年经历40多次大地震不倒，不是因为它比钢结构更硬——恰恰相反，是因为它"让"了。不是死扛，是让。让不是放纵——让是有设计的退让，在退让中把外部压力变成内部连接力。

Harness的核心法则——"Harness内Agent自主，Harness外人类签核"——就是这个缝。
Harness是榫卯，因为它精确地做到了榫卯的三重力学特性：有缝设计对应Agent自主空间，碰壁收紧对应Oracle和停止条件，木材弹性回弹对应Memory和进化飞轮。
R2-AIOS的架构用四个要素精确描述：Harness是确定性编排，Oracle是机械裁判，Memory是跨会话持久，Loop是跑到机械条件满足才停。

同一个Skill，在Harness治理下是IP，在个人工具场景下是便利贴。Skill的IP属性不是内生属性，是系统赋予的属性。
今天的遭遇：修好的bug明天又冒出来，强调不能改的状态明天又被改掉，说不能删的老链路后天直接被清理。
那一刻才明白，AI coding最大的坑不是AI不会写代码，而是你没有告诉他什么不能写、什么不能改、什么不能碰。

那么，这个"缝"具体是怎么工作的？Harness在其中扮演什么角色、不扮演什么角色？答案出人意料：Harness的核心秘密，恰恰是它不执行。

> 出处：861-榫卯结构 harness+loop工程.md、528!!!!r2-aios.md、566Harness Engineering驾驭工程-AI Coding四层治理.md、786-skill的本质.md

## §03 卯口不执行

Harness的核心秘密是不执行。
它不做AI决策——不判断"这个Skill好不好"，只判断"这个Skill的输出是否符合契约"。这是有意为之的缝隙：如果Harness也要做判断，它就变成了另一个LLM，系统就失去了机械裁判。
它也不做EDA执行——不运行仿真、不跑综合、不求等价。它只做信号转换：把EDA的高熵原始输出（log、波形、error）转换为低熵结构化信号（key_metrics + error_summary + convergence_status）。

这不作为，恰恰是榫卯有缝设计的工程表达。
卯就是凹进去的孔洞。Harness就是那个孔洞——它定义了一个空间：编排规则（agent / parallel / pipeline / while）、停止条件（MAX_ITER / 无进展检测 / 预算上限）、签核边界（Harness内Agent自主，Harness外人类签核）。它是一个"凹"的形状——不是实心的控制块，而是空心的界定的运行空间。

在这个空间内，Agent自由行动：自己读代码、跑测试、改文件、验证结果。小偏差时Agent自转修复——发散时回Agent自转，人不介入。Agent在Harness里跑Loop，产出结果后交给Oracle判——发散回Agent自转，收敛全绿则人签核。

Oracle不是LLM，是机械裁判——Spike diff、arch-test签名、Formal等价、DRC clean、timing WNS大于等于0。
停止条件不可协商：连续3轮随机差分0发散、官方一致性测试全绿且签名与Spike逐字一致、post-P&R阶段DRC clean加timing WNS大于等于0、人类签核记录落盘。
Harness的编排不是硬连线。动态工作流让Harness能在运行时实时编写编排脚本，启动子Agent舰队并行作战。Skill可以随时替换、增删，Harness不改一行代码。

不是Harness预先堵死了所有错误路径——那是钉子的逻辑。Harness的逻辑是：平时给你空间，但碰壁的那一刻，壁是硬的，判断是机械的，停止条件不可协商。
没有Harness的Agent，就像没有卯口的榫头——到处乱撞。566文件的亲身经历：AI越强，项目越崩，因为"你没有告诉它什么不能碰"。这正是卯口缺失的后果。
AI负责思考与行动，机器负责裁判，人类负责品味与签核。三者各干各的——其余全是浪费。

壁是硬的，但壁不是一开始就紧的。外部压力越大，壁收得越紧——这就是收窄决策空间的机制。

## §04 收窄决策空间

Harness的逻辑是：平时给你空间，但碰壁的那一刻，壁是硬的，判断是机械的，停止条件不可协商。
越压越紧的本质是：外部不确定性越大，Harness的门控越严密。

外部压力增大时，Harness通过三道机制收窄决策空间。
任务越复杂，Oracle的标准不降，但Loop跑更多轮。
Agent跑偏，停止条件触发——MAX_ITER硬编码退出、无进展连续N轮熔断、预算耗尽告警。
模型幻觉，Maker-Checker分离——做工作的模型不等于判断工作完成的模型，独立Agent审查。

三个硬性停止条件是任何生产级loop的必选项，不是可选项。
第一个是最大迭代次数，MAX_ITER硬编码退出条件。
第二个是无进展检测，连续N次无进展则loop终止并告警。这不是优化，是熔断。
第三个是预算上限——Token上限、金额预算。这不是小气，是工程纪律：一个没有天花板的loop，就是一张没有额度上限的信用卡。
没有机械Oracle，循环必须停止——否则是幻觉加速器。

于是任务被分成四层，Tier 0文档样式注释，Tier 1普通bug小功能，Tier 2 API DTO mapper数据契约，Tier 3认证状态机AI解析历史兼容链路。
门控梯度随之收紧：Tier 0最松，Tier 3最死。
Oracle是机械裁判——Spike diff、arch-test签名、Formal等价、DRC clean、timing WNS大于等于0。
应县木塔近千年经历40多次大地震不倒，靠的是榫卯的间隙设计和负反馈自锁。R2-AIOS在AI快速迭代、模型频繁更换、Skill持续进化的环境中保持稳定，靠的是Harness的契约设计和门控梯度。

从那以后，AI不能上来就写代码，它必须先回答：这是什么问题？属于哪一层风险？会影响哪些链路？哪些地方不能动？改完怎么验证？出事怎么回滚？
不是Harness预先堵死了所有错误路径——那是钉子的逻辑。Harness的逻辑是：平时给你空间，但碰壁的那一刻，壁是硬的。

碰壁收紧之后呢？钉子被撞了就松了，焊缝被压了就裂了。但榫卯不同——木材被撑变形后会弹回去。这个回弹力，才是榫卯防松的真正秘密。工程中如何复现这种回弹力？

## §05 回弹力的工程复现

木头被撑变形后想弹回去——这个回弹力才是榫卯防松的真正机制。不是钉子那种摩擦力，是材料本身的结构弹性。
循环工程的分水岭不是"能不能让AI自己跑"，而是跑完以后，系统有没有记住成功、理解失败、自主进化。

在工程中，回弹力的复现机制是Memory乘以Loop乘以Skill蒸馏构成的进化飞轮。
每次Loop执行产生trace，trace进入记忆——四层记忆架构从草稿纸到短期到长期到模型先验——失败模式被蒸馏成Skill，Skill进化，Harness在下一次执行时更强。

给AI看完整的原始执行轨迹（Trace），比只看300字摘要诊断更准，效果好百分之四十四。每一轮都能揪出至少一个之前没看见的问题。
四指令协作流构成一个增强飞轮：人工经验和失败案例进入蒸馏环节，蒸馏后的Skill经记忆写入持久化，Agent用Skill执行任务，验证指令在上线前把关，进化指令在线持续优化。优化产生的负反馈和正信号回灌到蒸馏环节——新一轮的Skill比上一轮更强。
Harness设计水平上升，AI可控性上升，产出质量上升，度量可信上升，投入信心上升。这个飞轮一旦转起来，系统就不只是在完成一次任务——它在让下一次更好。

张三用Agent调通了一个偶发失败的测试，花了4小时试了5条错路。李四第二天遇到了几乎一样的问题——从零开始。张三的经验留在他的私人聊天记录里，对李四没有任何帮助。
两个团队都跑CI修复loop。A团队的loop有15个Skills文件，涵盖API约定、错误处理模式、部署流程。B团队的loop没有Skills——Agent每次修复CI失败都要重新猜测。三个月后，A团队的loop修复准确率百分之九十二，B团队百分之六十一。差距不在模型，在Skills。

失败了不再是代价，是写入系统记忆的生产性资产。系统在给自己长骨头。
Loop Engineering的起点不是自动化，是记忆。没有记忆的循环，只是昂贵的重复。
越压越紧，天生防松。用木材的弹性和几何形状，把外部压力变成连接力。

至此，我们看到了两个独立的力量：Harness定义了卯口形状，Loop定义了榫头运动。但两者之间的关系不是加法——是乘法。当两者协同，会产生一种更深刻的几何效应。

## §06 两层合力燕尾效应

驾驭工程等于设计卯口形状的工程学——决定边界在哪、空间多大、壁在哪里。
循环工程等于设计榫头如何在卯口里运动的工程学——决定循环怎么跑、什么时候停、跑偏了怎么纠正。
两者的关系不是加法，是乘法。

燕尾榫——形状头大身小，像楔子——越压越把卯口的两边撑开。木头被撑变形后想弹回去，回弹力反过来把榫头死死夹住。越压越紧，天生防松。
这个几何效应对应Harness乘以Loop的乘性协同：卯口形状设计和榫头运动方式设计互相放大。

榫头本身由三个要素构成，三者是乘法关系，任一项为零整体为零。Skill是榫头上的具体几何形状。CLI是榫插入卯的通道和接口。LLM是木材本身的弹性模量——LLM的推理能力就是木头的弹性，它能变形、能适应、能在缝里找到咬合位置。模型越强，弹性越好，能适应的卯口形状越复杂。
三者乘性关系的具体表现：Skill好但CLI缺失，则榫头形状完美但插不进去。CLI齐全但LLM弱，则通道在但材料没弹性，一压就断。LLM强但没有Skill，则弹性好但没有形状，在卯口里打滑。

不止榫头三要素是乘法关系。R2-AIOS的四件套也是乘法——Harness乘以Oracle乘以Memory乘以Loop。任一项为零，整体为零。缺Oracle等价于高速幻觉加速器，缺Memory等价于每日新实习生，缺Loop等价于知识积累无交付边界。

## §07 不是死扛是让

驾驭工程的核心设计哲学不是"控制每一步"，而是"定义边界加留出缝隙加碰壁收紧加弹性回弹"。不是死扛，是让。让不是放纵——让是有设计的退让，在退让中把外部压力变成内部连接力。

要理解这个哲学，需要回到榫卯的力学本质。
小力时，缝起作用。榫头在卯口里微微活动，不硬碰硬。地震来了，结构可以晃动、滑移，把冲击能量变成摩擦热散掉——不是死扛，是让。
大力时，缝让榫头挤上卯口壁。榫头越往里顶，接触面越大。是燕尾榫——形状头大身小，像楔子——越压越把卯口的两边撑开。木头被撑变形后想弹回去，回弹力反过来把榫头死死夹住。越压越紧，天生防松。

Harness不是钉子。钉子靠的是贯穿——硬打进去，靠摩擦力死撑。钉子结构的系统是硬编码规则、if-else流程、固定脚本。Prompt Engineering时代写的"万能模板"就是钉子——一次性钉死，不给退让空间，久了就松。Harness恰恰相反：不预控Agent每一步，而是定义边界加留出缝隙加碰壁收紧加弹性回弹。

不把签核下放——Harness内Agent自主，Harness外只有人类taste。
Harness的核心法则——"Harness内Agent自主，Harness外人类签核"——就是这个缝。这三道缝，就是榫卯的"间隙"。它不是缺陷，是设计。
AI会写代码，但它不懂边界。AI会完成任务，但它不懂代价。AI会优化逻辑，但它不懂世故。AI coding最大的坑不是AI不会写代码，而是你没有告诉他什么不能写、什么不能改、什么不能碰。

循环工程的分水岭不是"能不能让AI自己跑"，而是跑完以后，系统有没有记住成功、理解失败、自主进化。没有记忆的循环，只是昂贵的重复。失败了不再是代价，是写入系统记忆的生产性资产。系统在给自己长骨头。

经验都可以被蒸馏，但信任只能落在人身上。AI可以执行，机器必须裁判，人要保留签核。
驾驭工程是设计卯口形状的学问——决定空间多大、壁在哪里、什么时候收紧。用木材的弹性和几何形状，把外部压力变成连接力。

这就是驾驭工程的本质：用Agent的弹性和Harness的几何形状，把外部压力变成内部连接力。不是死扛，是让。让不是放纵——让是有设计的退让，在退让中把外部压力转化为连接的紧固力。

芯片TDD：测试先行是第一性原理

芯片验证有三个问题，不解决就会持续恶化。

第一，认知过载。写RTL时大脑同时处理三件事——功能对不对、结构好不好、边界条件漏没漏。工作记忆容量有限，三件事一起想，必然顾此失彼。复杂模块里的并发竞争和流水线耦合，就是这样漏掉的。

第二，恐惧与熵增。每次修改都可能引入副作用。时间越长，副作用越多，系统越脆弱。你越怕改，越不敢改；越不敢改，代码越腐烂。这是死循环。

第三，歧义与延迟。自然语言写的规格天然带歧义。"输入被认为是Active则设置标志"——什么算Active？当拍还是下一拍？这些歧义在传统流程里要等到调试甚至流片后才暴露。而在千行代码里找Bug和在十行增量里找Bug，成本差距不是线性的，是指数级的。

三个死结，一个对策：把验证顺序反过来。先定义什么是对的，再实现。

TDD的核心不是多写测试，而是认知分离。把同时压在大脑上的三件事拆成三步。RED阶段只做分析——写断言，定义期望行为，不想实现。GREEN阶段只做直觉——用最简单的方式让测试通过，不想优化。REFActor阶段只做审美——优化结构，不改行为。三步每步只占一个认知槽位。

TDD在芯片验证中从不以原始形态出现。它化身为九种方言：断言先行是SVA属性版的test-first，Shift-Left是测试前置的项目级放大版，BDD/Gherkin是团队协作层面的可执行规格，CI/CD是红绿循环的工程底座——没有秒级反馈，TDD循环就无法运转。

工具决定方法论的采纳速度。当SystemVerilog没有原生测试框架时，TDD采纳率趋近于零。当Python工具链（pytest+cocotb+pyuvm）成熟后，工程师不需要任何TDD培训就能自然采用测试先行——pytest本身就是TDD的标准框架。工具易用性驱动方法论采纳，而非理念认同。

TDD的终极形态是AI自动化闭环。LLM生成候选规格，形式验证工具执行等价检查作为"测试运行"，失败时自动解析反例并反馈迭代。Red-Green-Refactor循环从人的纪律变成了机器的本能。

没有精确的Spec就写不出精确的断言，写不出精确的断言就过不了RED阶段。这不是理念偏好，是工程纪律。复杂度越高，TDD相对于传统方法的优势越大——因为复杂度放大了认知过载、恐惧熵增和歧义延迟。

### R2七条_swiss.png
R2 7 条：把 AI 跑成流水线

1 组织记忆回写
教会后面的会话，别只快一个人——每次从空白提示词开始，公司只是「一批更快的个人」，没有组织复利。

2 验证左移 + 砍流程
编码不再是瓶颈；AI 做第一道自动检查，人把关安全、品味与签核，最嘈杂、最痛苦的工作流优先自动化或砍掉。

3 分层上下文工程
像带实习生：先地图、再规矩、再接现场；工具是感官不是插件，入口控工具数量、出口控 token 体积。

4 交整段流程，不是单动作
智能体要丢有头有尾的整体活——判断标准：这是不是一件能从头到尾交付的事。

5 高价值问题 × Loop × 客观验收
批量撒高价值问题并行跑；验收绑死 EDA 结果或行业标准打分——机器裁判先于人的品味争论。

6 串行 → 并行的重构识别模式
识别哪些步骤必须串行、哪些可 fork 成独立 Loop、哪些可 Worktree 并行——否则只会堆 Agent，不会提速。

7 大目标先行 + 放手
一直盯一步改一步，智能体就退化成聊天框；先连滚带爬跑完全链路，再往下拆、再并行、再蒸馏。

收束：4 交什么 → 3 怎么准备 → 5-7 怎么跑 → 2 怎么验 → 1 怎么留
