# R2-第二刀：特性内短闭环，机械 Oracle 作裁判

没有第二刀，团队仍停留在「官知」：RTL 仿真、人肉看波形、人 review，一次改动常见要耗掉三天。第二刀把这件事拆成三层，不要混为一谈。

## 三层

### 动作层 Andy
在每个特性内裁 MVP，分成必要、增强、未来，并建短自验环。「必要」必须是完整切片，不是做一半。

### 裁判层 Vicky
用确定性 Oracle——Python golden、CRC、diff——而不是 AI 自评。Layer 1 参考答案，Layer 2 统计模型，Layer 3 形式化等价，Layer 4 波形断言。

### 节奏层 CI/CD
分钟级反馈与 CI/CD 门禁，三类场景用机械规则判定。Sprint 0 通常只需前两层。

## 六种路径差异

③最依赖「高层描述 / 生成物 / Oracle」三者同源，否则分钟级反馈只是在加速制造漂移。②可以先有 Python 参考答案再逼近 RTL。④⑤的 Oracle 往往要同时覆盖硬件行为与驱动可见行为。①⑥若厂商已提供硅验证向量，第二刀可以薄。

> 「庖丁不用眼睛逐寸看——第二刀用机械 Oracle 替代人肉判断，官知止而 Oracle 行。」

## 第三刀：ICD 冻结后的端到端拼合

第三刀处理的是信息源不唯一。Word、RTL、UVM 配置三份文档三个版本，每次集成都要人肉对齐。答案不是再多写一份文档，而是 ICD 与 r2-specx 同源——意图、测试、RTL 从同一事实源投影出来，机器可读。

### Pipe
什么数据在流，吞吐、反压、FIFO 如何约束

### Valve
怎么控制和观察，开关、配置、状态读回

### Translator
两边协议是否一致，不一致时谁来做位宽、协议与 CDC 翻译

ITRS 想要的是从规格到完成系统的集成设计流，并且每一步都能被验证。第三刀用发布火车把它落到可演示的站：

### R1
Boot 与 UART 拼合，对齐波特率与 GPIO 冲突，所有特性的「必要」拼成可演示价值链

### R2
加深 LDPC、Channel、HARQ，对齐 LLR 格式与 FIFO

### R3
放压力、安全、优化等「未来」项

DoD：上电 → LDPC 译码 → HARQ NACK → 重传 → Pass。纵向切片不是横向分组：增量交付，不靠末期大返工。机器对齐之后，对齐损耗不应随特性数平方增长；每一批都要能端到端演示。

> 「庖丁沿筋骨主腱走——第三刀让接口沿 ICD 中轴对齐，拼合后善刀而藏之。」
