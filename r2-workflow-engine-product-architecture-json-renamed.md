# R2-工作流引擎：产品架构图

让每一次“完成”都可验证、可追溯、可恢复。

## L3 用户入口

CLI 终端 | IDE 插件 | API 调用 | CI/CD 触发器
统一进入工作流引擎。

## L2 任务分类器：入口路由

输入：用户请求 + 工作区快照
输出：匹配最优执行模式

任务分类引擎：请求特征提取 → 模式匹配 → 路由分发

路由类型：goal-led 多步骤任务 | debug 调试 | review 审查 | quick 快速执行 | light 纯问答

## L1 执行引擎：三条主状态机

### 目标驱动模式
建计划文件 → 逐目标执行 → 每个目标需证据 → 最终验证

### 调试模式
复现问题 → 多假设竞争 → 证伪测试 → 验证修复

### 审查模式
记录发现清单 → 逐项解决 → Gate 检查

## L0 状态账本：文件驱动持久化

- plan.json：计划文件 · 任务序列、依赖关系、完成状态
- review.json：审查清单 · 问题记录、严重等级、关闭状态
- trail.jsonl：轨迹日志 · 每步操作记录、命令+输出、时间戳

## 验证门 Gate

通过条件：
1. 任务分类正确
2. 所有目标有 ledger.jsonl 证据记录
3. 验证命令已执行：测试 / lint / 类型检查
4. review.json 中无 open 状态发现

任一条件不满足 → 阻塞完成声明 → 输出剩余风险清单

### 输出结果
- 通过：完成报告 · 改了什么、验证了什么、剩余风险
- 阻塞：风险清单 · 缺失证据、开放发现、补救建议

## 产品定位
不是“AI 做对”的工具。
是“AI 做完有证据才过”的工程基础设施。
与 TDD / CI / Code Review 同构：不保证质量，只阻止不合格的声明。

## 核心价值
决策者：完成后有可审计证据链；中断后可恢复；不绑定模型或平台
工程师：不靠感觉做完；Gate 是机械判据；Evidence Log 是工程记忆

## 在 R2-AIOS 全栈中的位置
- L3 上层应用：specx / c2rtl / topgen / verismart / buildsoc，解决“做什么”
- L2 AIOS 内核：4角色 · 精益 · 架构 · 验证 · 工作流引擎，解决“怎么做、怎么停、谁签核”
- L1 基础设施：DataLabel · TDD · Specx · ConfDrivenGen · EDA-CLI，解决“数据从哪来、工具怎么接”

## 一句话
一个文件驱动的 Agent 状态机：请求进来先分类，按模式执行，每步留证据，Gate 拦截虚假完成，输出可验证报告。不碰模型，不绑平台，不说没证据的话。
